## Context

当前日报新闻段采用“尽可能多抓取”的策略，导致：
1) 同源新闻数量过大（可达 100+），可读性差；
2) 聚合站/导航页掺杂，重要新闻被淹没；
3) 突发财经、宏观、地缘线索无法稳定置顶。

系统已具备 RSS、搜索、FT 回退三层能力，因此本次设计重点是“排序与筛选引擎”而不是新增数据源。

## Goals / Non-Goals

**Goals:**
- 让新闻段稳定输出高价值条目（默认 15 条）
- 突发、宏观、地缘新闻可解释地优先
- 保留容灾（RSS/搜索失败时仍有回退）
- 修复债券利差展示单位误导（bp vs %）

**Non-Goals:**
- 不引入付费新闻 API 依赖
- 不做全文抓取与摘要生成（仅标题+短摘要）
- 不做复杂机器学习模型（规则评分优先）

## Decisions

1. **规则评分优先（可解释）**
   - 评分维度：分类权重 + 来源权重 + 域名权重 + 突发关键词 + 宏观/地缘关键词 + 时效分。
   - 原因：当前目标是快速提升信号质量，规则可快速迭代且日志可解释。
   - 备选：LLM 逐条打分（成本高、时延高、可控性差）。

2. **“阈值 + 限额”双层筛选**
   - 先按最低分过滤，再按来源上限/分类上限控制结构均衡。
   - 原因：避免单一来源刷屏，也避免某个主题（如美股）完全挤占宏观/地缘条目。
   - 备选：纯 Top-N（会被单源占满）。

3. **去重采用标题指纹 + URL 归一化**
   - 原因：不同渠道常有同文异链或异文同链，单一策略不够稳。

4. **收益率曲线单位显式化**
   - `macro_monitor` 的利差是 bp，报告中统一显示为 `bp`。
   - 避免将 `^IRX` 误作为 2Y 直接算 `%`，减少误导。

## Risks / Trade-offs

- [阈值过高导致漏报] → 增加“补齐”逻辑，条数不足时从次高分回填。
- [关键词规则偏置] → 保留配置化入口，后续可迁移到 `config/news.yaml`。
- [源站质量波动] → 采用多源聚合与来源权重，降低单源异常影响。
- [实时性与稳定性冲突] → RSS 主源时效好但有噪声，搜索补充提高覆盖；两者并行后再筛选。

## Migration Plan

1. 先在 `news.py` 增加评分+筛选，不改外部接口（返回结构向后兼容）。
2. 更新 `builder.py` 以适配利差单位和条目精简后的展示。
3. 本地运行 `market_daily_run.py --output report` 回归验证。
4. 观察 1-2 天输出质量，再决定是否将规则迁移到配置文件。
